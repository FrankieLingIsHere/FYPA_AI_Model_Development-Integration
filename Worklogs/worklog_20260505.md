# Worklog - 2026-05-05

## Summary of Work
Today focused on final CASM state-machine hardening across Local Mode provisioning, report-origin routing, local/cloud metrics, realtime notifications, and local Gemma/Ollama tolerance on CPU-only devices.

## Key Accomplishments

### 1. Provisioning Active-State Recovery
- Fixed the post-heartbeat provisioning state so `active` is treated as a terminal success state instead of falling back to `submitted`.
- Made repeated provisioning/checkup requests idempotent for already-active devices.
- Added missing-identifier recovery for incognito/session-loss cases: if a fresh active local heartbeat already exists, `/api/provision/request` returns `active` instead of creating a new pending/rejected device.

### 2. Report Routing and Sync Tags
- Added frontend report-origin routing so `[Cloud]` and `[Local Synced]` reports open through the cloud backend even while the active app mode points at `localhost`.
- Kept unsynced `[Local]` reports routed to the local backend.
- Updated report image URLs, prefetch, status polling, and report-open actions to use the report row’s `source_scope` rather than the global mode.
- Preserved `sync_local_cache`/`local_cache_sync` markers so synced local reports remain tagged as `[Local Synced]` after upload to Supabase.

### 3. Unified Metrics
- Updated frontend stats logic for Local Mode to fetch cloud and local violation lists with `Promise.allSettled`, merge by `report_id`, and calculate cumulative dashboard/analytics totals from the unified dataset.

### 4. Local Notification Latency
- Forced Local Mode realtime to use the local SSE stream instead of Supabase realtime when the active backend is `localhost`.
- Added immediate backend realtime rows after YOLO capture and metadata creation, before Gemma/Ollama report generation begins.
- This decouples the “violation detected” UI/voice notification from slow local report generation.

### 5. Local Gemma/Ollama CPU Tolerance
- Increased the local forced Ollama NLP read timeout default from 120s to 240s so CPU-only Gemma runs are less likely to fall into the fallback block.
- Kept notifications decoupled so longer CPU inference does not block live-monitoring alerts.
