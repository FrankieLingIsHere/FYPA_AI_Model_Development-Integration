# Worklog - 2026-05-06

## Summary of Work
Today focused on the remaining CASM PWA transition bugs: offline cloud-report crashes, stale Local Mode fallback state leaking into Cloud Mode after reconnect, local Gemma/Ollama fallback observability, and the local metrics/tag mapping drift between local and cloud reports.

## Key Accomplishments

### 1. PWA Offline Cloud Report Handling
- Updated the Service Worker to recognize report document and report-details API requests.
- Added graceful offline responses for cloud report routes so failed Railway fetches return a controlled unavailable payload instead of crashing the PWA.
- Added frontend report-open guarding so offline cloud report clicks show: "Cloud report details are unavailable while offline."
- Added cached report HTML fallback so previously prefetched cloud reports can still open offline from browser storage.
- Added an explicit report HTML cache flow: reports are prefetched into IndexedDB with a stable `report-html:<backend>:<report_id>` key, cached HTML gets a `<base>` tag for original backend-relative assets, and offline report opening checks this cache before showing the unavailable toast.
- Expanded cache warming so ready cloud/shared/local-synced reports are prefetched after violation rows load, not only after a user clicks a report. The Reports page also trickle-warms up to 60 ready report HTML files per load while online.
- Tightened `[Local Synced]` handling so synced local reports with local artifacts open through the local backend while offline instead of being blocked as cloud-only.
- Added a Service Worker runtime API-cache clear message for network/mode transitions.

### 2. Mode Transition Leak Cleanup
- Added explicit cloud-transition cleanup for stale offline/fallback state.
- Cleared transient local/session flags such as offline mode and NLP failure markers before switching back to Cloud Mode.
- Forced backend base URL re-resolution on reconnect so the frontend does not keep using stale localhost/offline routing.
- Hooked the browser online event into the same cleanup path before auto-sync and cloud-mode recovery continue.

### 3. Gemma/Ollama Fallback Mystery Instrumentation
- Added detailed VLM route logging in `caption_image.py` for Ollama/Gemma preflight, model availability, provider attempts, cache hits, response timing, output length, and preview text.
- Added detailed NLP route logging in `report_generator.py` for provider selection, strict local fallback settings, Ollama model readiness, successful JSON parsing, and deterministic fallback application.
- The logs now distinguish three cases clearly: cache hit with no provider execution, actual Ollama/Gemma execution, and hardcoded fallback block usage.

### 4. Local Metrics and PPE Tag Mapping
- Patched local filesystem report-row collection to preserve `missing_ppe`, `ppe_tags`, YOLO detection tags, `violation_count`, `person_count`, and summaries.
- Added dashboard stats aliases so Local Mode can read report totals through the same field names the frontend expects.
- Updated the Analytics page to normalize the same report-count aliases and display `Reports Generated` as a first-class metric.
- Expanded frontend local report normalization to parse YOLO tags from direct fields, metadata, detection arrays, and summary text.
- Updated the reports list to render specific PPE labels for local reports instead of the generic "PPE Violation Detected" text.

### 5. Auto-Sync and Local Synced UI State
- Routed local-cache Supabase sync through the local backend so reconnect sync scans the user's actual local violation folders instead of accidentally posting to the cloud backend.
- Added `origin: local_synced` / `source_scope: synced_local` to reconnect sync requests and backend sync metadata.
- Returned exact queued report IDs from `/api/reports/sync-local-cache`.
- Added frontend sync update events so reports immediately flip from `[Local]` to `[Local Synced]` without a reload.
- Kept half-generated local reports out of the immediate sync/tag flip: reports without `report.html` now continue local generation first, then become sync candidates on the next reconnect/reconcile pass once ready.
- Registered a Background Sync tag as a browser-supported safety net, with the Service Worker messaging open clients to run local report sync on reconnect.
- Pushed a realtime row after backend local-cache sync reconciliation so SSE clients can also repair the tag.
