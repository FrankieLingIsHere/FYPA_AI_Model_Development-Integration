## Follow-up: Cloud Reprocess Snap-back

- Reproduced the latest stuck-report behavior against the deployed backend with report `20260516_112819`.
- Confirmed `POST /api/report/20260516_112819/generate-now` accepted a cloud-scoped reprocess request, but immediate status/list reads still returned stale local metadata.
- Root cause:
  - stale `synced_local` / local-handoff metadata was being interpreted before cloud artifact evidence in cloud mode
  - manual reprocess did not persist the repaired source scope onto the existing violation row before the page refreshed
  - browser-side local draft/list/status caches could still rehydrate the old local row after a successful cloud fallback
- Backend fix:
  - cloud mode now repairs invalid stale `synced_local` metadata to `Cloud` when cloud artifacts exist and there is no strict local origin
  - manual cloud reprocess now persists repaired `detection_data` immediately, preserving detection details while removing stale local handoff markers
- Frontend fix:
  - successful cloud fallback/reprocess removes the matching local draft
  - list caches for the same report are repaired to `source_scope=cloud` / `source_label=Cloud`
  - stale per-report status/detail caches for the same report are cleared so refreshes cannot snap back to local state
- Performance check:
  - ran `cProfile` on the focused backend report-status contract
  - confirmed the measured time is dominated by application/import startup, not the new source-scope repair path
